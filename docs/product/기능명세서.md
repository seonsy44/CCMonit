# Claude Code Monitor 기능명세서

## 1. 기능 범위 요약
본 문서는 Claude Code Monitor의 기능 요구사항을 정의한다.

핵심 기능은 아래 8개 범주로 구분한다.
1. 세션 모니터링
2. 계층 구조 추적
3. 토큰/비용 추적
4. 로그 저장 및 조회
5. 파일 활동 추적
6. 알림 및 이상 징후 탐지
7. 검색/필터/정렬
8. 리포트 및 내보내기

---

## 2. 기능 요구사항

### FR-01 세션 연결 및 감시
- 시스템은 Claude Code 세션 시작을 감지할 수 있어야 한다.
- 시스템은 현재 세션의 시작 시각, 경과 시간, idle 시간, 작업 디렉터리, 세션 ID를 표시해야 한다.
- 시스템은 단일 세션 기준 실시간 화면을 제공해야 한다.
- 시스템은 데이터 소스 연결 상태(Adapter health)를 표시해야 한다.

### FR-02 계층 구조 표시
- 시스템은 최소한 다음 계층을 지원해야 한다.
  - Session
  - Team(선택)
  - Agent
  - Task
  - Skill Invocation
  - Tool Call
- 상위/하위 관계가 확인되지 않는 이벤트는 orphan으로 분류해 별도 표시해야 한다.
- 각 노드는 상태, 시작 시각, 마지막 업데이트 시각, 누적 시간, 로그 수를 가져야 한다.

### FR-03 현재 활성 작업 표시
- 시스템은 현재 running/waiting/idle 상태 작업을 우선적으로 노출해야 한다.
- 작업 목록은 최근 활동 순 또는 시작 시각 순으로 정렬 가능해야 한다.
- 사용자는 접기/펼치기로 계층 깊이를 조절할 수 있어야 한다.

### FR-04 스킬 및 툴 사용량 집계
- 시스템은 작업별 스킬 호출 수를 집계해야 한다.
- 시스템은 툴 종류별 호출 수를 집계해야 한다.
- 예: Bash, Read, Write, Grep, AskUserQuestion, WebFetch 등
- 시스템은 전체 세션 합계와 현재 화면 구간 합계를 모두 보여줄 수 있어야 한다.

### FR-05 토큰 추적
- 시스템은 가능한 경우 다음 값을 저장해야 한다.
  - input tokens
  - output tokens
  - cache read tokens
  - cache write tokens
  - total tokens
- 토큰은 최소 다음 단위로 집계해야 한다.
  - session
  - agent
  - task
  - tool call(가능한 경우)
- 각 토큰 수치에는 accuracy status(exact/derived/estimated/unavailable)가 있어야 한다.
- 추정값은 exact 값과 시각적으로 구분되어야 한다.

### FR-06 시간 추적
- 시스템은 작업별 시작/종료/경과 시간을 기록해야 한다.
- 시스템은 마지막 이벤트 이후 idle 시간도 계산해야 한다.
- 시스템은 장시간 미변경 작업을 stuck 후보로 분류할 수 있어야 한다.

### FR-07 작업 로그 저장
- 시스템은 모든 작업에 대해 개별 로그 스트림을 저장해야 한다.
- 로그는 최소 다음 이벤트를 포함할 수 있어야 한다.
  - 상태 변경
  - 스킬 시작/종료
  - 툴 호출 시작/종료
  - 파일 변경
  - stderr 발생
  - 사용자 질문 발생
  - 요약/결과 출력
- 로그는 작업 단위로 열람 가능해야 하며 시각 순 정렬을 지원해야 한다.

### FR-08 파일 활동 추적
- 시스템은 파일 생성/수정/삭제/이동 이벤트를 감지해야 한다.
- 파일 활동은 가능하면 관련 task/agent와 연결해야 한다.
- 최근 파일 활동 패널에는 시각, 파일 경로, 이벤트 유형, 관련 작업을 표시해야 한다.
- 동일 파일이 짧은 시간 안에 반복 수정되면 강조 표시할 수 있어야 한다.

### FR-09 알림 및 이상 징후
- 시스템은 다음 상황을 알림 대상으로 지원해야 한다.
  - 일정 시간 이상 업데이트 없음
  - stderr 연속 발생
  - 토큰 급증
  - 재시도 반복
  - orphan 이벤트 증가
  - 데이터 어댑터 연결 끊김
- 알림은 화면 상단 또는 별도 패널에서 확인할 수 있어야 한다.
- 알림은 확인/해제/무시 설정이 가능해야 한다.

### FR-10 태스크 상세 보기
- 사용자는 특정 태스크를 선택해 상세 화면을 볼 수 있어야 한다.
- 상세 화면에는 다음이 포함되어야 한다.
  - 기본 정보
  - 상태 이력
  - 관련 agent/team
  - 호출한 skill 목록
  - 툴 호출 타임라인
  - 파일 변경 목록
  - 토큰 사용량
  - stderr / warning / summary 로그

### FR-11 세션 상세/요약 보기
- 사용자는 전체 세션 기준 요약 화면을 볼 수 있어야 한다.
- 세션 요약에는 다음이 포함되어야 한다.
  - 총 에이전트 수
  - 활성 태스크 수
  - 완료/실패 태스크 수
  - 스킬 종류별 사용량
  - 툴 종류별 사용량
  - 토큰 총합
  - 파일 변경 수
  - 경고 수

### FR-12 필터/검색/정렬
- 사용자는 상태, 에이전트명, 태스크명, 툴 종류, 파일 경로, 시간 범위 기준으로 필터링할 수 있어야 한다.
- 사용자는 태스크명/파일명/로그 내용 검색을 수행할 수 있어야 한다.
- 정렬 기준은 최근 활동, 토큰 사용량, 소요 시간, 오류 수를 지원해야 한다.

### FR-13 리포트 및 내보내기
- 세션 종료 후 요약 리포트를 생성할 수 있어야 한다.
- 내보내기 포맷은 최소 JSON, Markdown, CSV를 지원하는 것이 바람직하다.
- 리포트에는 Top N 토큰 소비 작업, 실패 작업, 오래 걸린 작업, 파일 변경 상위 목록이 포함되어야 한다.

### FR-14 설정
- 사용자는 refresh interval을 변경할 수 있어야 한다.
- 사용자는 로그 저장 레벨(raw/full/summary)을 선택할 수 있어야 한다.
- 사용자는 토큰 추정 허용 여부를 설정할 수 있어야 한다.
- 사용자는 민감정보 마스킹 규칙을 설정할 수 있어야 한다.

### FR-15 종료 및 복구
- 시스템은 비정상 종료 후에도 가능한 범위에서 이전 세션 로그를 복구해야 한다.
- 시스템은 최근 세션 목록을 제공해야 한다.
- 세션 재열기 시 read-only 회고 모드를 지원하면 좋다.

## 3. 비기능 요구사항

### NFR-01 성능
- 기본 refresh interval은 2초를 목표로 한다.
- 이벤트 수집은 UI 갱신과 분리되어야 한다.
- 대량 로그가 들어와도 화면이 심하게 끊기지 않아야 한다.

### NFR-02 확장성
- 새로운 툴 종류가 추가돼도 집계 체계가 깨지지 않아야 한다.
- 어댑터를 교체해도 도메인 모델은 유지되어야 한다.

### NFR-03 신뢰성
- 파싱 실패 이벤트도 raw 형태로 보존해야 한다.
- 일부 어댑터가 실패해도 전체 화면이 완전히 죽지 않아야 한다.

### NFR-04 사용성
- 키보드만으로 주요 기능 조작이 가능해야 한다.
- 기본 화면은 한 번에 현재 상태를 파악할 수 있게 요약 중심이어야 한다.
- 과도한 색 사용보다 상태 구분과 계층 구조 인지가 우선되어야 한다.

### NFR-05 보안/프라이버시
- 로그 저장 시 민감 문자열 마스킹 옵션을 제공해야 한다.
- 사용자가 원하지 않으면 raw input/output 본문을 저장하지 않을 수 있어야 한다.

## 5. 집계 규칙 제안
1. total_tokens는 가능하면 원천 이벤트 값을 우선 사용한다.
2. 원천 total이 없으면 부분값 합산으로 계산한다.
3. exact와 estimated 값이 섞인 집계는 화면에 혼합 상태를 표시한다.
4. 상위 노드 집계 시 하위 노드 중복 합산이 일어나지 않도록 event_id 기반 dedupe가 필요하다.
5. long-running task는 현재까지 누적값과 종료 후 확정값을 구분한다.

## 6. 로그 저장 정책 제안

### 로그 레벨
- summary: 핵심 메타데이터만 저장
- normal: 이벤트 요약 + 주요 payload 일부 저장
- full: raw event/raw text까지 모두 저장

### 저장 포맷
- JSONL 권장
- 이벤트 발생 순서 보장을 위해 monotonic sequence 부여 권장
- 태스크 로그와 세션 로그를 병행 저장 권장

### 보존 전략
- 최근 N개 세션은 빠른 로컬 조회용 인덱스 유지
- 오래된 raw 로그는 압축 보관 옵션 제공

## 8. 수용 기준(예시)
- 사용자가 기본 화면에서 현재 활성 태스크 목록을 볼 수 있다.
- 특정 태스크를 선택하면 상세 로그와 토큰 정보를 확인할 수 있다.
- 세션 종료 후 Markdown 요약 리포트를 생성할 수 있다.
- exact token이 없을 때 estimated 표시가 명확히 드러난다.
- 최근 파일 변경 20건을 관련 태스크와 함께 확인할 수 있다.
- stuck 후보 태스크가 시각적으로 강조된다.
