# Claude Code Monitor 기획서

## 1. 문서 목적

이 문서는 Claude Code 기반 하네스 환경에서 **세션, 에이전트, 팀, 태스크, 스킬, 도구 호출, 파일 변경, 토큰 사용량**을 한눈에 파악할 수 있는 터미널 모니터링 도구 **Claude Code Monitor(가칭: CCMonit)** 의 기획 방향을 정의한다.

본 문서는 “지금 이 순간 무엇이 일어나고 있는가”를 빠르게 파악하는 운영용 관점과, “어떤 작업이 얼마나 많은 토큰과 시간을 소비했는가”를 추적하는 분석용 관점을 동시에 다룬다.

---

## 2. 문제 정의

Claude Code를 하네스 환경에서 운용할 때 다음과 같은 문제가 반복된다.

1. 세션 안에서 많은 일이 동시에 일어난다.
   - 여러 에이전트가 동시에 뜬다.
   - 팀/태스크/스킬이 중첩되어 실제 진행 상황을 따라가기 어렵다.
   - 지금 활성 상태인 작업과 이미 끝난 작업의 구분이 어렵다.

2. 로그는 남지만 “운영 관점에서 읽기 좋은 구조”가 없다.
   - stdout/stderr, 텍스트 로그, 파일 변경, 툴 호출이 서로 흩어져 있다.
   - 나중에 보면 어떤 로그가 어느 태스크의 결과인지 연결하기 어렵다.

3. 토큰 전략을 세우기 어렵다.
   - 도구별/작업별/에이전트별로 얼마나 토큰을 썼는지 모른다.
   - 입력과 출력 중 어디서 소모가 큰지 파악하기 어렵다.
   - 재시도, 루프, 장문 컨텍스트 누적 등 비용이 커지는 패턴을 발견하기 어렵다.

4. 장시간 실행 중인 작업의 상태 파악이 어렵다.
   - 현재 진행 중인지 멈췄는지 모른다.
   - 파일은 바뀌는데 UI 상으로는 체감되지 않는다.
   - 병렬 작업이 많아질수록 체감 난이도가 급격히 올라간다.

## 3. 제품 목표

### 3.1 핵심 목표

- **실시간 가시성**: 현재 세션에서 어떤 작업이 진행 중인지 빠르게 파악한다.
- **구조화된 추적성**: 세션 → 팀 → 에이전트 → 태스크 → 스킬/툴 호출 흐름을 구조적으로 본다.
- **토큰/비용 분석**: 작업 단위, 도구 단위, 에이전트 단위의 토큰 사용량을 추적한다.
- **사후 회고 가능성**: 특정 작업 하나를 나중에 다시 열어도 “무슨 일이 있었는지”를 재구성할 수 있어야 한다.
- **터미널 친화성**: TUI 기반으로 빠르고 가볍게 사용 가능해야 한다.

### 3.2 성공 기준

- 사용자가 모니터 화면만 보고도 **현재 활성 작업 3~5개를 5초 안에 설명할 수 있다.**
- 세션 종료 후 **어떤 작업이 토큰을 많이 사용했는지 상위 원인을 바로 찾을 수 있다.**
- 태스크 로그를 열었을 때 **시작 시각, 종료 시각, 관련 파일, 호출한 스킬/툴, 토큰 사용량**을 한 번에 볼 수 있다.
- 장시간 세션에서도 “멈춤 / 과도한 재시도 / 토큰 급증 / 오류 반복” 같은 이상 징후를 인지할 수 있다.

## 4. 대상 사용자

- Claude Code를 단독으로 오래 사용하는 개발자
- 에이전트/팀/스킬이 많은 하네스를 운용하는 개발자
- 여러 하위 에이전트를 병렬로 돌리는 사용자
- 비용과 토큰 전략을 최적화하려는 사용자
- 실험 세션을 회고하고 재현 가능한 기록으로 남기려는 사용자

## 5. 제품 컨셉

Claude Code Monitor는 단순 로그 뷰어가 아니라 **“운영 대시보드 + 작업 단위 감사 추적기 + 토큰 분석기”** 를 결합한 도구다.

핵심 컨셉은 다음 세 가지다.

1. **지금 보기**
   - 현재 살아 있는 세션/에이전트/태스크를 즉시 보여준다.
   - 무엇이 active / waiting / idle / failed 인지 구분한다.

2. **흐름 보기**
   - 어떤 상위 작업이 어떤 하위 작업을 만들었고, 그 안에서 어떤 스킬과 툴이 실행되었는지 연결해서 보여준다.

3. **소모 보기**
   - 시간, 토큰, 파일 변경, 오류 수, 재시도 수를 함께 보여줘서 “비용이 왜 발생했는지”를 해석할 수 있게 한다.

## 7. 핵심 사용 시나리오

### 시나리오 A. 병렬 하위 에이전트 감시

사용자는 메인 세션에서 여러 하위 에이전트를 띄운다. 모니터는 현재 살아 있는 하위 작업들을 목록으로 보여주고, 각 작업이 어떤 역할인지(예: 성능 평가, 테스트 품질 점검, 가독성 점검)를 표시한다. 사용자는 어떤 작업이 오래 걸리는지, 어떤 작업이 이미 끝났는지, 어떤 작업이 토큰을 많이 먹는지 즉시 파악한다.

### 시나리오 B. 토큰 전략 회고

세션이 끝난 뒤 사용자는 “왜 이번엔 토큰이 많이 들었지?”를 확인하고 싶다. 모니터는 작업별 입력/출력 토큰, 도구 호출 수, 캐시 사용 여부, 재시도 수를 보여준다. 사용자는 특정 태스크가 지나치게 큰 컨텍스트를 사용했다는 사실을 찾아낸다.

### 시나리오 C. 문제 작업 추적

사용자는 특정 태스크가 중간에 멈춘 것 같은 느낌을 받는다. 모니터는 해당 작업이 마지막으로 로그를 남긴 시각, 마지막 툴 호출, 최근 파일 변경, stderr 발생 여부를 보여준다. 사용자는 “실제 멈춤”인지 “긴 계산 중”인지 구분할 수 있다.

### 시나리오 D. 파일 변경 중심 추적

사용자는 지금 어떤 파일이 누가 만든 변경인지 알고 싶다. 모니터는 최근 파일 활동 목록을 보여주고, 각 변경을 관련 태스크/에이전트와 연결한다. 사용자는 “이 파일은 테스트 에이전트가 건드렸고, 저 파일은 리팩터링 태스크에서 수정했다”를 확인할 수 있다.

## 8. 설계 원칙

1. **지금 일어나는 것**을 우선 보여준다.
2. 정보량이 많아도 **구조화된 계층**으로 압축한다.
3. 숫자는 많아도 **해석 가능한 지표** 위주로 보여준다.
4. 토큰 수치는 가능하면 exact를 쓰되, 불가능하면 **estimated / unavailable 상태를 함께 표시**한다.
5. 모든 중요한 이벤트는 **나중에 태스크 단위로 복기 가능**해야 한다.
6. 터미널 환경에서 **낮은 인지 부하**를 유지해야 한다.

- queued
- running
- waiting
- idle
- done
- failed
- cancelled
- unknown

### 9.3 토큰 정확도 상태

- exact: 런타임/로그에서 직접 제공
- derived: 여러 이벤트를 합산해 계산
- estimated: 모델/문자수 기반 추정
- unavailable: 데이터 소스 부재

## 10. 수집 전략 제안

Claude Code 및 하네스의 내부 이벤트 포맷은 환경마다 다를 수 있으므로, 수집부는 **어댑터 구조**로 설계하는 것이 좋다.

### 10.1 추천 어댑터

1. **Runtime Adapter**
   - Claude Code 프로세스를 래핑해 시작/종료/표준출력/표준에러/하위 프로세스를 관찰
2. **Harness Adapter**
   - 팀/에이전트/태스크/스킬 메타데이터를 하네스에서 직접 수신
3. **Log Tail Adapter**
   - JSONL 또는 텍스트 로그를 tail 하며 이벤트 파싱
4. **File Watch Adapter**
   - 작업 디렉터리 파일 생성/수정/삭제 감시
5. **Git Adapter**
   - 현재 브랜치, 변경 파일 수, staged/unstaged 상태 추적
6. **Token Estimator Adapter**
   - 정확한 토큰이 없을 때 추정값을 생성

### 10.2 왜 어댑터 구조가 중요한가

- 하네스 포맷이 바뀌어도 화면 계층은 유지할 수 있다.
- 추후 다른 AI CLI에도 일부 재사용 가능하다.
- exact / estimated 데이터가 섞여도 동일한 집계 모델을 유지할 수 있다.

## 11. 추가 제안 기능

아래 기능들은 사용자가 직접 언급하지 않았지만 실제 운영에서 유용성이 높다.

### 11.1 이상 징후 감지

- 일정 시간 이상 로그가 없는 태스크 표시
- 짧은 시간 내 재시도 반복 감지
- 토큰 급증 경고
- stderr 폭증 경고
- 동일 파일을 여러 태스크가 경쟁적으로 수정하는 상황 감지

### 11.2 효율성 지표

- 태스크당 시간 대비 토큰 효율
- 파일 1개 수정당 토큰 사용량
- 성공/실패 대비 토큰 사용량
- 캐시 적중률 또는 캐시 활용 추정

### 11.3 작업 분류 자동화

- planning / coding / refactor / test / review / docs 등 태스크 유형 추정
- 어떤 유형의 작업이 토큰을 많이 먹는지 비교

### 11.4 세션 회고 리포트

- 상위 토큰 소비 작업 Top N
- 가장 오래 걸린 작업 Top N
- 실패한 작업 목록
- 가장 많이 건드린 파일 목록
- 세션 타임라인 요약

### 11.5 비밀정보 보호

- 로그 저장 시 경로/키/비밀 문자열 마스킹 옵션
- 전체 raw prompt 저장 여부를 설정화
- 민감한 출력은 본문 대신 해시/요약만 저장하는 옵션

## 13. 리스크와 대응

### 리스크 1. 정확한 토큰이 항상 나오지 않음

대응:

- accuracy status를 반드시 표기
- exact / derived / estimated 구분 표시
- 리포트에서 추정치와 실측치를 분리

### 리스크 2. 로그 포맷이 자주 바뀜

대응:

- 이벤트 표준화 레이어 도입
- adapter 별 파서 테스트 준비
- unknown event도 raw 보존

### 리스크 3. 정보가 너무 많아 화면이 복잡해짐

대응:

- 기본 화면은 요약 중심
- 상세는 drill-down 방식
- 필터/접기/정렬 제공

### 리스크 4. 로그 저장이 과도하게 커짐

대응:

- raw 저장 레벨 선택
- 압축/보존 기간 정책
- 대형 payload는 요약 저장 옵션 제공

## 14. 기대 효과

- Claude Code 기반 복잡한 하네스 작업을 “운영 가능한 상태”로 만든다.
- 비용과 토큰 전략을 감이 아니라 데이터로 세울 수 있다.
- 병렬 에이전트 작업의 디버깅과 회고가 쉬워진다.
- 장시간 실행 세션에서 불안감과 관찰 비용을 크게 줄일 수 있다.
