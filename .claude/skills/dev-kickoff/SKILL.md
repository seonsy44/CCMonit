# dev-kickoff

이 스킬은 **어떤 개발건을 할지 아직 불분명할 때** 쓴다. 프로젝트 상태를 분석하여
가능한 개발건 후보를 도출하고 `.claude/scratch/dev-list.md`에 추가한다.

- `dev-kickoff` = "어떤 개발건을 할까" (개발건 레벨 탐색)
- `dev-open` = "이 개발건에서 어떤 스프린트를 할까" (스프린트 계획 고정)

**언제 쓰나:**
- 무엇을 할지 감이 안 잡힐 때
- 여러 방향이 있어서 비교하고 싶을 때
- 특정 주제에 대해 가능한 개발건을 정리하고 싶을 때

**언제 안 쓰나:**
- 이미 dev-list에 개발건이 있으면 바로 `dev-open {slug}`으로 간다
- 개발건 단위 관리 없이 바로 실행하려면 `dev-sprint`으로 직행한다

## 입력

- `$ARGUMENTS`는 optional.
  - (없음) — 프로젝트 전체 상태에서 개발건 도출
  - `[주제]` — 해당 주제 범위 내에서 개발건 도출
  - `[주제] [부가적요청사항]` — 주제 + 추가 조건으로 개발건 도출

입력 예시:
  - (없음) — 현재 프로젝트 상태 기반 전체 탐색
  - `도메인 레이어` — 도메인 관련 개발건만
  - `TUI 빠르게 동작하게` — TUI 관련 + 속도 우선 조건
  - `다음 마일스톤 준비` — 로드맵 기반 개발건 도출

- 각 개발건의 `next-prompt.md`를 훑어보기

주제 파라미터가 있으면:
- 해당 주제와 관련된 docs, 코드를 추가로 읽는다

## 해야 할 일

### 0단계: interview (조건부)

주제 파라미터가 있지만 모호한 경우에만 실행한다.

모호함 판단 기준:
- 주제가 2단어 이하이고 프로젝트 내 대응 범위가 3개 패키지 이상에 걸치는 경우
- 부가적요청사항이 상충하거나 해석이 2가지 이상인 경우
- "잘", "빨리", "좋게" 같은 방향만 있고 구체 대상이 없는 경우

모호하다고 판단하면:
- 1~2개 질문만 한다 (선택지 형태 권장)
- 답변 후 1단계로 진행한다

모호하지 않으면 바로 1단계로 간다.

### 1단계: 개발건 후보 도출

1. 프로젝트 현재 상태를 짧게 진단한다.
2. 가능한 개발건 후보를 2~5개 도출한다.
   - 각 개발건은 `dev-open`에 slug 파라미터로 줄 수 있는 단위다.
   - 각 개발건 안에는 예상 스프린트 2~3개를 포함한다.
3. 각 개발건에 대해 판단한다:
   - 긴급도 (urgent / normal / low)
   - 불확실성 (높음 / 보통 / 낮음)
   - 선행 조건 (없음 / 있으면 명시)
4. `.claude/scratch/devs/`에 진행 중인 개발건이 있으면 "이어가기" 후보로 포함한다.

### 2단계: dev-list.md에 추가

1. `.claude/scratch/dev-list.md`가 없으면 새로 생성한다.
2. 도출한 개발건 후보를 기존 목록에 **추가**한다.
   - 기존 항목은 유지한다.
   - 동일 slug가 있으면 해당 행을 갱신한다.
3. "프로젝트 상태 요약"과 "최종 갱신" 날짜를 갱신한다.

## 출력 형식

**프로젝트 상태 요약**: 1~3문장

**새로 추가된 개발건 후보:**

| # | 개발건명 | slug | 예상 스프린트 | 긴급도 | 불확실성 | 선행 조건 | 비고 |
|---|----------|------|---------------|--------|----------|-----------|------|
| 1 | ... | ... | 스프린트1, 스프린트2 | ... | ... | ... | ... |

**추천**: 1순위 개발건과 짧은 이유

**다음 단계**: `/dev-open {slug}`으로 선택한 개발건을 시작한다.
